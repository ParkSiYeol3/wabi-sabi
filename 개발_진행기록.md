### 39단계 — Supabase 기반 rate limit 저장소 (#187, PR #188, 2026-07-18)
- 기존 rate limit(`lib/rate-limit.ts`)은 Upstash Redis(미설정)→인메모리 폴백 2단. **인메모리는 서버리스 인스턴스별 독립(Map)이라 Vercel 에서 상한이 인스턴스 수만큼 느슨하고 콜드스타트마다 리셋.** 외부 인프라(Redis) 없이 인스턴스 간 공유되는 상한을 Supabase 테이블로 확보.
- **마이그 0029**: `rate_limits` 테이블(RLS on·정책 없음 → anon/authenticated 직접 접근 전면 차단, service_role 만 우회) + `check_rate_limit` RPC(원자적 `upsert` 고정창 카운터, `security definer`, execute=`service_role`만). 낮은 확률(0.5%)로 1시간 지난 창 청소. **마이그 0030**: `window_start` 단일 인덱스(청소 DELETE 가 PK 선두 bucket 없이 전수 스캔하던 것 해소, CodeRabbit).
- **백엔드 3단화**: Redis → **Supabase** → 인메모리. Vercel 은 `SUPABASE_SERVICE_ROLE_KEY` 있고 Upstash 없어 자동으로 `supabase` 선택. 원격 실패 시 인메모리 fail-open(부가 기능이 레이트리밋 장애로 죽지 않게).

- **CodeRabbit 리뷰**: `adminConfigured()` 가 키만 검사해 `NEXT_PUBLIC_SUPABASE_URL` 빠진 배포에서 매 요청 인메모리 폴백하던 것 → URL 도 검증하도록 수정.

  - **판별 규칙**: 홈(`searchParams` prop → 선-dynamic)·`[id]`(dynamic route, generateStaticParams 없음 → 빌드 프리렌더 안 함)는 자동 안전. **순수 정적 라우트**에서 cached-public 쿼리를 쓰면 `export const dynamic = "force-dynamic"` 필요. 쿼리는 `unstable_cache` 라 dynamic 렌더여도 DB 왕복은 그대로 없다(성능 이득 유지, 빌드는 DB 비의존 → 더 견고).
  - 로컬 빌드는 `.env.local` 있어 프리렌더가 통과해버려 못 잡음 → **CI(env 없는 환경)가 이 클래스의 회귀를 잡는다.**
